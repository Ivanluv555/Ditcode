## 0. 文档目标
本文件用于沉淀第一阶段前端方案，统一说明：

- 我们讨论过并已采用的核心方案
- 当前技术选型与实现边界
- 当前项目架构与数据流
- 下一阶段计划与风险清单

---

## 1. 第一阶段达成目标（已落地）

### 1.1 数据与后端策略
- 采用 **Node.js mock 服务 + JSON 文件** 作为当前持久层。
- 前端不直接读写文件，只通过 `/api/*` 调用服务。
- 已接入认证与工作区持久化：
  - `mock-server/data/auth.json`
  - `mock-server/data/workspace.json`
  - `mock-server/data/community.json`

### 1.2 工作流与存档制度
- 核心模型从“单任务列表”升级为“**每用户多存档（archives）**”。
- 每个存档包含：
  - `messages`（对话）
  - `tasks`（mock 任务）
  - `modelAsset`（单作品，符合“一个存档一个建模作品”）
- 规则已更新为：
  - 点击“发起新工作”不立即落空档；
  - **首次提交输入并开始处理时才创建新存档**。

### 1.3 社区与分支（类 Git）
- 社区项目可被用户“打开并 fork”为自己的存档分支。
- 用户在 fork 分支上完成 remix 时，可回写社区 remix 计数。
- 用户删除仅影响自己存档，不影响社区原始项目。
- 存档默认私密，发布后才可见于社区。

### 1.4 页面交互与体验
- 首页欢迎态支持首访/新建触发，并避免残留。
- 主输入区支持“每次输入提交自动展开右面板”。
- 传图逻辑约束：**每个存档仅首轮输入可带图，后续隐藏入口并强制禁带图提交**。
- 设置弹层与用户弹层已改为方向化下拉动画（非简单遮罩弹窗）。

### 1.5 主题与视觉统一
- 亮/暗主题采用全局语义 Token（`global.css`）统一配色。
- 社区、我的内容、登录注册、工作区核心组件已接入变量化配色。

## 2. 已讨论方案与最终采纳结论

### 2.1 持久化方案
**讨论：** 本地内存 mock vs Node+JSON 持久化  
**采纳：** Node+JSON。原因：可跨刷新保留、可模拟真实接口边界、便于后续替换真实后端。

### 2.2 存档模型
**讨论：** 延续 `tasks/assetHistory` vs 引入 `archives`  
**采纳：** 引入 `archives`，保留 legacy 投影兼容旧页面。

### 2.3 社区交互
**讨论：** 直接编辑社区项目 vs fork 到用户空间  
**采纳：** fork 到用户存档组（类分支），社区保持原件稳定。

### 2.4 传图规则
**讨论：** 每轮都可上传 vs 首轮限定  
**采纳：** 首轮限定（满足后续模型输入约束，简化状态机）。

### 2.5 首页信息架构
**讨论：** 欢迎语常驻 vs 工作态对话视图  
**采纳：** 欢迎语用于首访/新建入口，进入工作后展示存档对话信息。

---

## 4. 当前架构（按职责）

## 4.1 前端目录分层
- `src/app`：应用编排（路由、壳层）
- `src/packages/*`：业务域模块（auth/workspace/discovery/content/settings）
- `src/shared/*`：跨域复用（样式、组件、webgl、工具）

## 4.2 后端 mock 分层
- `mock-server/server.js`：API 服务与数据校验/迁移
- `mock-server/data/*.json`：持久化数据

## 4.3 关键数据流
1. 用户操作（新建/输入/打开存档）
2. Pinia `useTaskStore` 更新归档状态
3. 调 `/api/workspace`、`/api/community` 同步
4. Node 服务写回 JSON
5. 页面按当前归档渲染

---

## 5. 当前主要数据结构（概念）

```json
{
  "version": 2,
  "activeArchiveId": "archive_xxx",
  "archiveOrder": ["archive_xxx"],
  "archives": {
    "archive_xxx": {
      "id": "archive_xxx",
      "title": "新工作",
      "ownerId": "user_xxx",
      "ownerName": "demo_user",
      "isPrivate": true,
      "sourceCommunityId": "",
      "remixCount": 0,
      "messages": [],
      "tasks": [],
      "modelAsset": null,
      "status": "idle",
      "createdAt": 0,
      "updatedAt": 0
    }
  }
}
```

说明：
- `modelAsset` 单对象，保证“一个存档一个作品”。
- `messages` 用于后续接模型上下文。
- `sourceCommunityId` 用于 fork/remix 追踪。

## 8. 建议验收口径（第一阶段）

- 新建后不落空档，首条输入触发建档。
- 打开社区项目会 fork 到用户存档。
- 用户删除存档不影响社区原件。
- 私密存档不出现在社区，发布后可见。
- 每存档首轮可带图，后续隐藏上传入口。
- 亮暗主题下页面与弹层配色一致。
