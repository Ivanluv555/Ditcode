# 前端特性文档

> 适用阶段：当前前端第一阶段原型
> 
> 目标：把已经讨论并落地的前端特性、展示顺序、动画逻辑、可替换位置，以及开发过程中踩过的坑统一沉淀，便于后续接真实模型、真实后端和真实社区内容时直接替换。

---

## 1. 文档目标

本文档不是用例表，也不是数据库设计，而是偏“前端表现层”的统一说明，关注以下四件事：

1. 页面是怎么展示的。
2. 用户每一步操作后，界面应该怎么动。
3. 哪些位置后续可以直接替换为真实能力。
4. 我们已经踩过哪些坑，后续不要再踩。

---

## 2. 当前前端整体特性概览

当前项目已经从单纯的页面原型，演进成“可持续工作流原型”。核心特性包括：

- 首次访问展示欢迎态。
- 用户点击“发起新工作”后进入新的工作上下文。
- 用户输入内容后，主面板进入“对话流”状态。
- 每个存档只有一条主建模作品（单作品单存档）。
- 每个存档仅首轮允许上传图片，后续隐藏入口。
- 右面板在用户输入后自动展开，用于承载工作区/建模态信息。
- 社区内容支持打开、fork、remix，并且与用户自己的存档隔离。
- 用户可以对自己的存档进行删除、发布、取消发布。
- 亮色 / 暗色主题均通过统一 Token 驱动。

---

## 3. 核心展示顺序

这一部分是前端最重要的“视觉顺序”，后续无论接真实模型还是接真实后端，都建议尽量保持。

### 3.1 首次访问

**目标体验：**
- 进入站点后，先看到欢迎语，而不是直接看到空白对话流或 threejs 面板。
- 欢迎态是“引导式”的，不是长期常驻态。

**顺序：**
1. 页面壳加载完成。
2. 首访标记未命中时，展示欢迎语。
3. 用户开始输入后，欢迎态退出，进入工作态。

### 3.2 点击“发起新工作”

**目标体验：**
- “发起新工作”不是立即创建空存档，而是进入一个新的工作上下文。
- 此时应再次展示欢迎语，给用户一个新的开始。

**顺序：**
1. 用户点击“发起新工作”。
2. 当前工作上下文被清空/切换。
3. 回到首页欢迎态。
4. 用户第一次真正输入并提交后，才建立该存档。

### 3.3 用户输入并提交

**目标体验：**
- 主面板不空闲，显示用户输入形成的气泡流。
- 用户提交内容后，右面板自动展开。

**顺序：**
1. 用户在底部输入框输入文本。
2. 如果是首轮，允许附带图片；如果不是首轮，隐藏图片入口。
3. 用户点击发送。
4. 系统创建/切换到当前存档，右面板自动展开。
5. 主面板切到对话流展示。

### 3.4 打开旧存档

**目标体验：**
- 打开旧存档后，不显示 threejs 空面板，而是直接显示该存档的最后对话。
- 页面内容顶部对齐，不再上下居中。

**顺序：**
1. 用户从左侧历史区点击旧存档。
2. 前端切换到该存档。
3. 主面板展示该存档的最近对话记录。
4. 右面板保持自动展开（如果该存档需要展示建模作品/工作区状态）。

### 3.5 打开社区项目

**目标体验：**
- 社区项目不是直接改原件，而是 fork 成自己的存档。
- 进入后应像“git 分支”一样继续编辑。

**顺序：**
1. 用户点击社区卡片。
2. 系统 fork 出用户自己的存档分支。
3. 新存档归入用户存档组。
4. 主面板展示该分支的对话记录。
5. 后续输入属于该分支的修正行为。

## 4. 动画逻辑规范

这一部分是前端的统一“动效语言”，后续新增页面建议复用，不要再各写各的。

### 4.1 欢迎态动画

**适用位置：** 首页 `HomePage.vue`

**效果：**
- 欢迎语从轻微位移和透明度变化进入。
- 标题、描述、快捷入口分层出现。

**建议节奏：**
- 欢迎标题先出现。
- 描述稍后出现。
- 引导按钮最后出现。

### 4.2 下拉 / 弹层动画

**适用位置：** 设置弹层、用户弹层、删除确认弹窗。

**效果：**
- 不是简单遮罩跳出，而是按锚点位置决定方向。
- 设置弹层从下往上，用户弹层从上往下。
- 弹层内部项按顺序滑入。

**规则：**
- 面板动画方向必须与触发按钮位置相关。
- 内部条目使用 stagger（错峰）出现。
- 动画时长统一，不要局部过快或过慢。

### 4.3 主面板气泡动画

**适用位置：** 首页工作态对话区。

**效果：**
- 用户输入后，以气泡形式进入主面板。
- 气泡从下轻微上浮进入。
- 如果是发布存档，气泡上方补充用户名标签。

**规则：**
- 气泡流是主面板默认工作态，不要让主面板显得空闲。
- 单条气泡不要占据太多横向宽度。
- 时间戳与气泡主体分层展示。

### 4.4 卡片动画

**适用位置：** 社区页、我的内容页卡片。

**效果：**
- 卡片列表采用网格 / 瀑布式入场。
- 每张卡片按 delay 顺序进入。
- Hover 时做轻微上浮与阴影增强。

**规则：**
- 同一页面内卡片动效保持一致。
- 避免过强的缩放和过大的位移。
- 卡片动效要服务于内容，不要抢内容。

### 4.5 页面整体节奏

**建议节奏：**
- 页面壳层先到位。
- 标题区先入场。
- 内容区后入场。
- 操作区最后出现。

### 4.6 GSAP 统一动效层

**适用位置：**
- `App.vue` 的侧栏首屏、底部 composer、设置 / 用户 / 删除确认弹层。
- `HomePage.vue` 的欢迎态、气泡流入场。
- `DiscoveryPage.vue` 的页面标题入场。
- `MyContentPage.vue` 的页面标题、统计条、卡片与空状态入场。
- `AssetCard.vue` 的卡片首屏入场。

**我们现在怎么用 GSAP：**
- 用 `gsap.fromTo()` 控制首屏入场、位移、透明度和缩放。
- 用 `gsap.timeline()` 组织分层节奏和 stagger 错峰。
- 用 `gsap.context()` 管理组件卸载清理，避免路由切换后残留动画。
- 用统一的 `prefers-reduced-motion` 降级策略，用户开启减少动态效果时直接简化为静态展示。

**GSAP 负责的内容：**
- 页面级 reveal / 入场顺序。
- 弹层进入 / 退出。
- 列表项错峰出现（stagger）。
- 欢迎态、气泡流、卡片流这类“内容驱动型”动效。

**继续保留 CSS / Transition 的内容：**
- hover、focus、按钮轻反馈。
- 主题色切换、边框、阴影等轻量过渡。
- 布局上简单的宽度 / padding 变化。

**治理原则：**
- 页面大动效统一走 GSAP，不再在每个组件里重复写 keyframes。
- 轻量交互保留 CSS，避免引入不必要的脚本复杂度。
- 统一使用 `prefers-reduced-motion` 作为动效兜底，不要在每个组件里重复判断。

## 6. 我们讨论过的“坑”

这一部分是最重要的经验沉淀，建议后续开发时优先看。

### 6.1 欢迎态残留

**现象：**
- 欢迎页状态没有及时清理。
- 刷新或回跳后依旧显示欢迎态。

**原因：**
- 早期通过 URL query 控制欢迎态，导致残留。

**解决：**
- 改为 session 级一次性触发。
- 首页进入时消费后立即清理。

### 6.2 白屏

**现象：**
- 打开浏览器后页面空白。

**原因：**
- 组件导入 / 导出不匹配。
- 顶层 async / 初始化时序导致根组件不渲染。

**解决：**
- 做构建检查。
- 避免根组件异步阻塞渲染。
- 统一导入路径与默认导出策略。

### 6.3 空存档污染

**现象：**
- 用户一点击“发起新工作”就生成一个空存档。

**问题：**
- 存档列表会出现很多无意义记录。

**解决：**
- 仅当用户第一次输入并开始处理时再创建存档。

### 6.4 首轮图片限制

**现象：**
- 图片上传入口在所有轮次都开放。

**问题：**
- 不符合当前“每存档首轮允许带图”的业务规则。

**解决：**
- 由存档当前消息数决定是否允许上传。
- 非首轮隐藏入口并在提交时强制忽略图片。

### 6.5 社区 fork / remix 语义混乱

**现象：**
- 社区内容、个人内容、原始内容之间边界不清。

**问题：**
- 删除权限、计数、发布状态容易互相污染。

**解决：**
- 社区项目视为只读原件。
- 用户打开即 fork 成个人存档。
- remix 通过事件/计数单独记录。

### 6.6 右面板展开逻辑不统一

**现象：**
- 有时输入后不展开，有时打开旧项目后不展开。

**解决：**
- 统一在“提交输入”和“打开项目”两个入口处理展开逻辑。
- 右面板状态以当前存档是否需要展示工作区为准。

### 6.7 样式与动效不统一

**现象：**
- 不同页面卡片、按钮、弹层的配色和动画节奏不一致。

**解决：**
- 所有主色、背景、边框、阴影统一走 `global.css` Token。
- 动画统一时长、easing、延迟节奏。

## 7. 页面级说明

### 7.1 首页 `HomePage.vue`

- 欢迎态：首访、新工作时展示。
- 工作态：展示对话气泡流。
- 右侧面板：在用户提交输入后自动展开。

### 7.2 社区页 `DiscoveryPage.vue`

- 展示公开项目。
- 点击卡片进入 fork 流程。
- 卡片封面使用统一占位策略。

### 7.3 我的内容 `MyContentPage.vue`

- 展示当前用户的存档分支。
- 支持发布 / 设为私密。
- 支持继续编辑、打开归档。

### 7.4 App 壳层 `App.vue`

- 负责全局布局、侧栏、右上角按钮、弹层、右面板展开。
- 统一处理新建、打开存档、打开社区项目、删除存档等入口逻辑。

---

## 8. 后续建议

1. 把欢迎态、工作态、存档态三个状态明确成 store 级别状态机。
2. 把“存档 = git 分支”语义继续往下细化，增加来源链路展示。
3. 把所有展示接口统一为 archive DTO，减少 legacy 投影依赖。
4. 将动效 Token 统一补全到文档，后续方便设计协作。
5. 接真实模型前，先补消息角色和流式状态字段。
6. GSAP 继续优先用于页面级入场和弹层编排，CSS 继续负责 hover 和轻过渡，避免动画方案再次分裂。

---

## 9. 结论

当前前端已经具备较完整的特性骨架：

- 有存档
- 有分支
- 有社区
- 有发布
- 有删除
- 有统一动效和主题
- 有向真实模型接入的空间

因此这份文档的作用，是把“我们已经做出来的前端特性”定下来，作为后续第二阶段开发和真实后端接入的统一依据。
