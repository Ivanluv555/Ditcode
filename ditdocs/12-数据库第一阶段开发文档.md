# 数据库第一阶段开发文档（ORM 建模落地期）

## 1. 阶段定位

第一阶段目标是完成关系模型与 ORM 映射落地，确立“后端驱动数据库”的主架构。

## 2. 阶段目标

1. 以 JPA Entity 建立核心数据模型（用户、会话、存档、社区、remix）。
2. 明确表结构与实体关系的一一映射。
3. 采用 code-first（`ddl-auto=update`）建立结构演进机制。

## 3. 架构原则（关键）

1. 数据库**没有与后端分立设计**，而是由后端 `Service -> Repository -> ORM` 驱动。
2. ORM 实体是结构主源，SQL 草案是参考，不是主事实来源。
3. 业务约束优先落在后端服务层，数据库负责持久化与关系完整性。

## 4. 已落地模型

| 实体 | 对应表 | 关系要点 |
|---|---|---|
| `UserEntity` | `users` | 用户主数据 |
| `SessionEntity` | `sessions` | `ManyToOne -> users` |
| `ArchiveEntity` | `archives` | `ManyToOne owner` + 自关联 `source_archive_id` |
| `ArchiveMessageEntity` | `archive_messages` | `ManyToOne -> archives` |
| `ArchiveTaskEntity` | `archive_tasks` | `ManyToOne -> archives` |
| `ArchiveModelAssetEntity` | `archive_model_assets` | `@MapsId` 实现一存档一资产 |
| `CommunityPublicationEntity` | `community_publications` | `OneToOne archive(unique)` |
| `RemixEventEntity` | `remix_events` | 关联 publication、remixer_archive、remixer_user |

## 5. 阶段交付

1. 全量 Entity 映射与 Repository 接口定义。
2. 统一命名规范（snake_case 表/字段，`*_id` 外键）。
3. 数据源配置与 JPA 自动演进配置落地。

## 6. 验收口径

1. 后端启动后可根据 Entity 自动创建/更新基础表结构。
2. 实体关系可覆盖当前业务聚合（workspace/community/auth）。
3. “一存档一作品”等结构性约束可通过映射表达并稳定持久化。

## 7. 阶段风险与输入

- 风险：过度依赖自动演进可能导致历史结构不一致，需在下一阶段补治理规范。
- 输入：后端业务写入链路与事务边界设计。
