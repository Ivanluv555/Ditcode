# 后端第三阶段开发文档（模型网关与稳定性治理期）

## 1. 阶段定位

第三阶段目标是补齐模型网关能力，形成“前端请求 -> 后端整形 -> 上游模型”的可运维链路。

## 2. 阶段目标

1. 提供统一模型生成入口，屏蔽上游差异。
2. 支持无上游配置时的降级返回，保障前端流程可持续。
3. 补齐文档与配置项，便于环境迁移和联调。

## 3. 核心实现

### 3.1 模型网关接口

- 入口：`POST /api/model/generate`。
- 请求 DTO：`taskId/archiveId/prompt/imageName/imageBase64`。
- 响应统一：`ok/provider/taskId/prompt/imagePreview/finishedAt/raw`。

### 3.2 上游调用与降级

1. 配置了 `model.gateway.upstream-url` 时，转发请求并解析响应。
2. 支持上游返回 `image/*` 或 `application/json` 两种模式。
3. 未配置上游时，返回 fallback 动态图片地址，保证主链路不中断。

### 3.3 配置与运维

- `model.gateway.upstream-url`
- `model.gateway.api-key`
- `model.gateway.connect-timeout-ms`
- `model.gateway.read-timeout-ms`
- OpenAPI：`/api/docs`，Swagger UI：`/swagger-ui.html`

## 4. 阶段交付

| 交付项 | 说明 |
|---|---|
| 模型网关服务 | 上游适配 + 返回整形 + 统一错误语义 |
| 降级策略 | 上游不可用时提供 fallback 结果 |
| 配置化能力 | 关键超时与鉴权参数可环境覆盖 |
| 接口文档化 | API 可通过 OpenAPI 直接查看 |

## 5. 验收口径

1. 模型生成接口在“有上游/无上游”场景均能返回可消费结果。
2. 上游异常时返回明确业务错误，不暴露内部堆栈细节。
3. 前端可用同一响应字段处理任务完成态与结果展示。

## 6. 现阶段边界

- 鉴权目前以业务层 `requireCurrentUser` 为主，后续可进一步前移到过滤链。
- 模型请求字段约束可继续按前端真实输入策略细化（例如纯文本场景与图片字段关系）。
