# 前端第一阶段开发文档（原型闭环期）

## 1. 阶段定位

第一阶段目标是先把“可连续使用的创作原型”跑通，优先验证业务流程，不依赖真实后端。

## 2. 阶段目标

1. 搭建可运行的前端工程与基础路由。
2. 完成登录、工作区、社区、我的内容、设置等核心页面原型。
3. 用 Mock 持久化验证“存档（archive）驱动”的业务模型。

## 3. 技术与架构基线

- 技术栈：Vue 3 + Vite + Pinia + Vue Router。
- 前端分层：`app`（编排）、`packages`（业务域）、`shared`（复用层）。
- Mock 服务：`mock-server/server.js` + `mock-server/data/*.json`。

## 4. 关键实现

1. 形成 `archives` 聚合模型，统一承载 `messages/tasks/modelAsset`。
2. 落地“首条输入触发建档”，避免空存档污染。
3. 建立“每存档首轮可带图”规则，后续输入转纯文本修正。
4. 打通社区打开即 fork、个人继续 remix 的交互语义。
5. 引入全局主题 Token，完成亮暗主题基础统一。

## 5. 阶段交付

| 交付项 | 说明 |
|---|---|
| 前端页面骨架 | 登录、注册、首页、社区、我的内容、设置、工作台 |
| 存档化状态模型 | Pinia 内统一管理 archive 生命周期 |
| Mock API 联调 | 通过 `/api/*` 读写本地 JSON 数据 |
| 第一版交互规范 | 欢迎态、工作态、卡片态的切换路径可复用 |

## 6. 验收口径

1. 能创建并切换多个存档，刷新后仍可恢复。
2. 社区项目可打开并形成个人分支，不污染原项目。
3. 私密与发布状态在页面层可正确区分。
4. 核心页面在亮/暗主题下可用且风格一致。

## 7. 阶段风险与输入

- 风险：Mock 与真实后端契约可能漂移，需在第二阶段做接口收敛。
- 输入：统一接口字段、鉴权策略、错误模型，为后端替换做准备。
