# 后端架构与模块设计（ditserver）

## 1. 总体分层

```text
com.design26.ditserver
  core/      # 全局能力
  module/    # 业务模块
```

### `core`

- `config`：Security、CORS、Jackson、模型配置绑定
- `auth`：Bearer Token 提取与当前用户解析
- `error`：统一异常模型与全局异常处理
- `util`：通用 ID 生成器
- `controller`：健康检查等全局接口

### `module`

- `auth`：用户与会话
- `workspace`：档案聚合根（消息、任务、模型资产）
- `community`：发布、下架、remix
- `model`：模型网关（后端转发上游模型）

---

## 2. 设计原则

1. 所有业务规则在 service 层执行（权限、状态约束、计数规则）。
2. controller 只做入参接收和响应编排。
3. repository 只处理持久化访问，不承载业务判断。

---

## 3. 模型网关（新增）

### 接口

- `POST /api/model/generate`

### 输入

- `prompt`
- `taskId`（可选）
- `archiveId`（可选）
- `imageName`（可选）
- `imageBase64`（可选）

### 行为

1. 后端整理前端请求格式。
2. 若配置了上游地址，转发到模型服务。
3. 若未配置上游，返回 fallback 动态资源 URL（保证前端流程可运行）。

### 配置

- `MODEL_UPSTREAM_URL`
- `MODEL_API_KEY`
- `MODEL_CONNECT_TIMEOUT_MS`
- `MODEL_READ_TIMEOUT_MS`

---

## 4. 安全策略

1. API 采用 Bearer Token（sessions 表）。
2. `/api/workspace*`、发布/下架/remix 等需要登录态。
3. 密码存储采用 BCrypt。

---

## 5. 资源职责

1. 后端不承载前端 icon 资源管理。
2. 后端仅返回业务数据与动态资源地址。
3. 前端负责 UI 资产与图标渲染。
