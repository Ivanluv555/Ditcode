# 后端第二阶段开发文档（业务域落地期）

## 1. 阶段定位

第二阶段目标是完成核心业务域实现，替换前端 Mock 数据源并稳定主业务链路。

## 2. 阶段目标

1. 完成认证、工作区、社区三大模块的接口与服务实现。
2. 把关键业务规则沉淀到 service 层统一执行。
3. 保障聚合写入、权限校验和社区操作一致性。

## 3. 模块实现

### 3.1 认证模块（`/api/auth`）

- `session`、`register`、`login`、`logout`。
- 密码使用 BCrypt。
- 会话 token 入库并与用户绑定。

### 3.2 工作区模块（`/api/workspace`）

- 获取、全量写回、重置工作区。
- 以 archive 为聚合根，维护 `messages/tasks/modelAsset`。
- 服务层校验“每存档仅首轮任务允许图片”。

### 3.3 社区模块（`/api/community`）

- 社区列表、发布、下架、remix。
- 仅 owner 可发布/下架。
- remix 同步写事件并累计计数。

## 4. 关键规则

1. 权限控制在服务层统一校验，不在 controller 分散处理。
2. 删除存档时联动删除任务、消息、资产、发布、remix 依赖。
3. 对前端保留兼容字段（`tasks/assetHistory` 投影）。

## 5. 阶段交付

| 交付项 | 说明 |
|---|---|
| 三大业务模块 | auth/workspace/community 全链路可用 |
| 业务规则固化 | 权限、状态、约束统一下沉 service |
| 聚合读写模型 | archive 聚合写入与读取响应标准化 |
| 社区机制 | 发布/下架/remix 具备可追踪数据 |

## 6. 验收口径

1. 前端可通过真实接口完成登录、创作、发布、remix。
2. 非 owner 无法发布或下架他人存档。
3. 非法工作区数据写入会被明确拒绝并返回错误信息。

## 7. 阶段风险与输入

- 风险：接口演进与前端状态模型耦合较深，需持续做契约对齐。
- 输入：第三阶段模型网关与上游服务接入方案。
