# 数据库第三阶段开发文档（演进治理与稳定性期）

## 1. 阶段定位

第三阶段目标是在 ORM + 后端驱动模式下，建立可持续的结构演进、性能与数据治理机制。

## 2. 阶段目标

1. 固化 schema 演进流程，避免随业务增长失控。
2. 补齐索引、删除策略、审计链路等稳定性能力。
3. 明确环境配置与运维边界。

## 3. 演进治理策略

### 3.1 结构演进

1. 以 Entity 变更为入口，遵循 code-first 演进。
2. 中高风险结构变更（字段类型、唯一约束、索引）先在测试环境验证。
3. 文档同步更新 `ditdocs` 与 `ditdbase/数据库设计.md`。

### 3.2 性能治理

- 优先治理查询热点索引：
  - `archives(owner_id, updated_at)`
  - `archive_messages(archive_id, created_at)`
  - `archive_tasks(archive_id, created_at)`
  - `community_publications(is_active, updated_at)`
  - `remix_events(publication_id, created_at)`

### 3.3 数据生命周期

1. 软删除字段（如 `archives.deleted_at`）用于业务可恢复与审计追踪。
2. 事件表（`remix_events`）保留行为轨迹，支撑统计与审计。
3. 聚合删除必须由后端统一执行依赖清理，不允许前端定义删除顺序。

### 3.4 环境与配置

- 当前运行采用 MySQL + JPA（`spring.jpa.hibernate.ddl-auto=update`）。
- 数据库连接配置由后端环境变量/配置文件控制，不由前端参与。

## 4. 阶段交付

| 交付项 | 说明 |
|---|---|
| 演进规范 | ORM 主源 + 文档同步 + 变更验证流程 |
| 性能基线 | 核心查询路径的索引治理建议 |
| 生命周期规则 | 软删、事件、依赖清理策略明确 |
| 环境边界 | 数据库由后端配置与驱动，前端零侵入 |

## 5. 验收口径

1. 数据结构变更可控，跨环境一致性明确。
2. 核心列表与聚合查询在数据增长后仍保持可用性能。
3. 数据删除、发布、remix 行为可追踪、可审计。
