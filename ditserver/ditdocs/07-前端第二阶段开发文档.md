# 前端第二阶段开发文档（后端替换与联调期）

## 1. 阶段定位

第二阶段目标是将前端从 Mock 驱动迁移到真实后端契约，形成可持续联调链路。

## 2. 阶段目标

1. 以 `VITE_API_BASE` 为入口接入真实 API。
2. 建立统一请求层、鉴权头注入和错误处理模型。
3. 完成工作区/社区/认证的真实数据闭环。

## 3. 核心改造

### 3.1 请求与环境

- 新增 `src/shared/utils/apiClient.js` 统一请求。
- 通过 `VITE_API_BASE` 支持本地/联调环境切换。
- 统一非 2xx 响应抛错并透传后端 `message`。

### 3.2 状态层

- `useAuthStore`：维护 token 与会话恢复逻辑。
- `useTaskStore`：完成工作区聚合读写、社区发布/下架/remix、模型生成入口。
- 引入写队列（`workspaceWriteQueue`）降低高频覆盖写冲突。

### 3.3 业务流程

1. 登录后获取用户态并附带 Bearer Token。
2. 工作区通过 `/api/workspace` 全量回写聚合状态。
3. 社区通过 `/api/community/*` 执行发布、下架、remix 计数。

## 4. 阶段交付

| 交付项 | 说明 |
|---|---|
| 统一 API 访问层 | 请求基址、Header、错误处理标准化 |
| 前后端联调链路 | 认证、工作区、社区三条主链路打通 |
| 归档兼容能力 | 保留 `tasks/assetHistory` 投影兼容旧视图 |
| 前端模块化重构 | 业务包按 `auth/workspace/discovery/content/settings` 收敛 |

## 5. 验收口径

1. 刷新后可恢复登录态并继续编辑当前存档。
2. 存档内容可正确落库并再次加载。
3. 发布/取消发布后社区可见性与计数一致。
4. API 异常可被前端正确提示，不出现静默失败。

## 6. 阶段边界与风险

- 当前后端鉴权依赖业务层 `requireCurrentUser`，前端仍需确保关键请求携带 Token。
- 认证接口口径需持续统一，避免历史 mock 逻辑残留导致误调用。
