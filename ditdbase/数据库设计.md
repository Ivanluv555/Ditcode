# 数据库设计

> 版本：v1（面向当前前端第一阶段业务）  
> 目标：从当前 JSON mock 平滑过渡到可生产的关系型数据库（建议 PostgreSQL）

---

## 1. 设计目标

- 支持 **每用户多存档（archive）**。
- 支持 **单存档单作品（model_asset）**。
- 支持社区发布/取消发布、fork、remix 计数。
- 明确权限：用户仅可删除/修改自己的存档。
- 与当前前端用例表一致（`前端业务用例表.md`）。

---

## 2. ER 关系（概念）

- `users` 1---n `archives`
- `archives` 1---n `archive_messages`
- `archives` 1---n `archive_tasks`
- `archives` 1---1 `archive_model_assets`
- `archives` 0---1 `community_publications`
- `community_publications` 1---n `remix_events`
- `archives` n---1 `archives`（`source_archive_id`，fork 来源）

---

## 4. 关键业务约束

1. **新建存档时机**：仅首条输入提交并进入处理后创建（无空档污染）。
2. **首轮传图约束**：每个存档仅允许首条输入带图，后续输入仅文本。
3. **删除权限**：仅 `archives.owner_id == current_user.id` 可删。
4. **社区可见性**：`archives.is_private = true` 时不可见；发布后在 `community_publications` 激活。
5. **分支语义**：社区打开不改原件，创建新 `archives` 且记录 `source_archive_id`。
6. **单作品约束**：`archive_model_assets.archive_id` 唯一。

---

## 5. 索引建议

- `archives(owner_id, updated_at desc)`：我的内容/历史列表。
- `archive_messages(archive_id, created_at asc)`：对话回放。
- `archive_tasks(archive_id, created_at desc)`：任务状态查询。
- `community_publications(is_active, updated_at desc)`：社区流。
- `community_publications(owner_id, created_at desc)`：个人发布管理。
- `remix_events(publication_id, created_at desc)`：remix审计。

---

## 6. 状态流转

## 6.1 存档状态（archive.status）
- `idle` -> `inferencing` -> `success`
- `idle` -> `inferencing` -> `failed`

## 6.2 可见性状态（archives.is_private）
- 默认：`true`
- 发布：`true -> false`（创建/激活 publication）
- 取消发布：`false -> true`（publication 置 inactive 或下架）

## 6.3 remix流程
1. 用户打开社区发布项目。
2. 创建 fork 存档（`source_archive_id` 指向原档）。
3. 用户生成并形成 remix。
4. 写入 `remix_events`，并更新 `community_publications.remix_count`。

## 8. 从 JSON 到关系库迁移策略

## 8.1 数据源
- `auth.json` -> `users`, `sessions`
- `workspace.json` -> `archives`, `archive_messages`, `archive_tasks`, `archive_model_assets`
- `community.json` -> `community_publications`, `remix_events(可选初始化为空)`

## 8.2 迁移步骤
1. 建表并准备唯一约束与索引。
2. 编写一次性导入脚本（按用户遍历 archives）。
3. 导入后做一致性校验：
   - 每 archive 是否有 owner
   - 是否违反单作品约束
   - publication 是否对应有效 archive
4. 上线双写期：保持 JSON + DB 并行写（短期）。
5. 切换读流量到 DB，保留 JSON 只做回滚备份。
6. 稳定后下线 JSON 写路径。

## 9. 未来扩展预留

- 多模型版本管理：`archive_model_versions`（如后续需要多作品版本）
- 团队协作：`archive_members`（共享存档）
- 消息附件：`message_attachments`（非首轮图或参考资料）
- 审计日志：`audit_logs`（发布/删除/权限变更）

---

## 10. 结论

当前业务已可以沉淀为稳定数据库模型：
- 以 `archives` 为核心聚合根；
- 以 `community_publications + remix_events` 承载社区分支关系；
- 通过明确约束实现“私密默认、可发布、可fork、可remix、可审计”。

该设计可直接支撑第二阶段真实后端与模型服务接入。
